## 0. Product summary

A "grow the network" lane, distinct from CAR-15's "tend the network you have": a scheduled job
searches LinkedIn (via `harvestapi/linkedin-profile-search`, actor C) for **PMs who joined a
high-priority target company in the last 90 days**, dedupes them against contacts and tombstones,
and surfaces them as *candidate contacts* — "3 new PMs just joined Qualtrics — want to add any?"
One click adds a candidate as a `prospect` contact (with auto-enrich); one click dismisses
forever. Per Dawson's explicit instruction, the **admin dashboard gets an on/off toggle** for the
feature, following the existing per-account scrape-control pattern (with the existing bulk
all-on/all-off rail).

## 1. Actor & query shape (verified against live input schema, 2026-07-10)

Actor: `harvestapi/linkedin-profile-search` (BRONZE pricing verified live):
- **Search page: $0.10, ≤25 short profiles.** In `profileScraperMode: "Short"` there are **no
  per-profile fees** — a run costs exactly $0.10 × pages.
- Full profile $0.004 / +email $0.01 — **not used here**; enrichment goes through the existing
  actor-A path after a candidate is added.

Per-company run input:

```jsonc
{
  "profileScraperMode": "Short",
  "currentCompanies": [company.linkedin_url],   // schema: "Provide full LinkedIn URLs"
  "functionIds": ["19"],                        // Product Management (verified enum)
  "recentlyChangedJobs": true,                  // joined within last 90 days (verified copy)
  "maxItems": 25,
  "takePages": 1
}
```

with `maxTotalChargeUsd: 0.15` (one page + margin). One page per company per cycle is deliberate.

**⚠️ Item-shape caveat (audit):** actor C's Short-mode output shape is **unverified in this repo**
— `ApifySearchProfileItem` (`client.ts:137-147`) is typed against actor B, and nothing consumes a
`position`/`currentPosition` field today. **Before finalizing the ingest wrapper, run one live
Short-mode query (~$0.10) and capture the real item as a test fixture.** `discovery_candidates.position`
is nullable; populate from whatever the real shape provides (`currentPosition[]`, `experience[0]`,
or headline fallback).

`recentlyPostedOnLinkedIn` (ticket's "worth exploring") is **out of scope** — noted for a future
iteration. **Eligibility constraint:** only companies whose `companies` row has a `linkedin_url`
can be queried (actor requires full LinkedIn company URLs); rows created name-only are skipped.

## 2. Data model (one migration, `2026071?_discovery_feed.sql`)

### 2.1 `users.discovery_enabled boolean NOT NULL DEFAULT false`

On **`public.users`**, following `20260710090000_apify_account_controls.sql`:
- `NOT NULL DEFAULT false` — **default OFF**: new paid spend, nobody's bill changes on deploy
  (CAR-26 "default off, admin grants" posture).
- **No GRANT statement** (audit A2): the foundation migration's column allowlist
  (`GRANT UPDATE (first_name, last_name, email, phone, updated_at)`) auto-excludes new columns.
- **WITH CHECK pin**: DROP + re-CREATE `users_update_own`, **re-listing all four pinned columns**
  — `status`, `apify_enrichment_enabled`, `diff_analysis_enabled`, `discovery_enabled` (the
  DROP/CREATE shape means a forgotten column silently loses its pin — audit's sharp edge).
- `COMMENT` documenting it as an admin kill switch.

### 2.2 `discovery_candidates`

```sql
CREATE TABLE discovery_candidates (
  id                bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id           uuid NOT NULL REFERENCES users(id) ON DELETE CASCADE,   -- public.users (convention)
  company_id        int NOT NULL REFERENCES companies(id) ON DELETE CASCADE, -- int4 (audit A3)
  linkedin_url      text NOT NULL,              -- canonical (lib/linkedin-url.ts)
  public_identifier text,
  name              text NOT NULL,
  headline          text,
  location          text,
  photo_url         text,
  position          text,                       -- nullable; see §1 item-shape caveat
  raw               jsonb NOT NULL,             -- short-profile item, audit/debug
  status            text NOT NULL DEFAULT 'new' CHECK (status IN ('new','added','dismissed')),
  added_contact_id  int REFERENCES contacts(id) ON DELETE SET NULL,          -- int4 (audit A3)
  first_seen_at     timestamptz NOT NULL DEFAULT now(),
  last_seen_at      timestamptz NOT NULL DEFAULT now(),
  UNIQUE (user_id, linkedin_url)
);
```

- RLS exactly like `scrape_runs`: enable RLS, `FOR SELECT USING (user_id = auth.uid())`, no
  insert/update policies (service-client writes only). Indexes `(user_id, status)` and
  `(user_id, company_id, status)`. No `updated_at` trigger (repo has no such convention).
- **Dismiss is sticky:** re-discovery only bumps `last_seen_at` (+ refreshes headline/position/
  photo), never resurrects status.

### 2.3 `target_companies.last_discovery_at timestamptz`

Rotation cursor; stamped per *company* (all scoped rows updated together — a company can have
one company-wide + N office-scoped rows since `20260710070000`).

### 2.4 `scrape_runs` changes

- `company_id int NULL REFERENCES companies(id)` — the run→company link (no meta/jsonb column
  exists; **int4**, audit A4). Used by ingest and the per-company in-flight guard.
- `mode` CHECK: drop + re-add `scrape_runs_mode_check` as `IN ('profile','email','resolve','discovery')`
  (precedent `20260710050000`).
- `trigger` CHECK: drop + re-add `scrape_runs_trigger_check` as
  `IN ('manual','enrich_on_save','cadence','discovery')` — **first-ever alter of this constraint**;
  the auto-generated name comes from the inline CHECK in `20260710010000`.
- New RPC `sum_scrape_spend_mode(p_user_id, p_since, p_mode)` — like `sum_scrape_spend` but
  filtered by mode; powers the discovery soft cap + the cadence decoupling (§3.5).

## 3. Service layer — `careervine/src/lib/apify/discovery.ts`

### 3.1 Client addition (`lib/apify/client.ts`)

`startProfileSearchRun({ companyLinkedinUrl, maxCharge, webhookUrl })` — async start of actor C
with the §1 input and the same terminal-event webhook wiring as `startProfileScrapeRun`. New
constant `PROFILE_SEARCH_ACTOR = "harvestapi/linkedin-profile-search"` in `constants.ts`
(distinct from the existing `PROFILE_SEARCH_BY_NAME_ACTOR`, which is actor B).

### 3.2 Company selection — `selectDiscoveryCompanies(service, userId, limit)`

From `target_companies` join `companies`:
- `is_targeted = true`, company `linkedin_url IS NOT NULL`;
- dedupe scoped rows to one entry per `company_id`;
- exclude companies queried within `DISCOVERY_MIN_AGE_DAYS = 30` (actor's 90-day window makes a
  monthly revisit lossless);
- exclude companies with a pending discovery run — in-flight guard keyed on the new
  `scrape_runs.company_id` (`.eq("status","pending").eq("mode","discovery")` → Set of company ids;
  the cadence `contact_ids` flatMap pattern does *not* transfer);
- order `last_discovery_at ASC NULLS FIRST`, then `priority_score DESC NULLS LAST` (nullable
  column — many rows may be NULL);
- take `limit` = `DISCOVERY_COMPANIES_PER_RUN = 5`.

### 3.3 Trigger — `triggerDiscoveryBatch(service, userId)`

Checks in order: global kill switch (`APIFY_SCRAPE_DISABLED`), `isApifyConfigured()`,
**re-check `discovery_enabled`** (fail-closed read, so a future non-cron caller can't bypass the
admin switch), discovery soft cap (`DISCOVERY_SOFT_CAP_USD = 2` — MTD via
`sum_scrape_spend_mode(..., 'discovery')` + in-flight discovery reserve), global
`MONTHLY_SCRAPE_CAP_USD = 10` hard cap via `getMonthlySpendUsd`. Then per selected company:
insert pending `scrape_runs` row (`mode:'discovery'`, `trigger:'discovery'`, `company_id`,
`contact_ids` defaults `'{}'`) → start actor run → stamp `target_companies.last_discovery_at`
for every scoped row of that company (stamped at trigger so a failed run can't cause same-week
re-spend; the 24h sweep keeps the ledger honest).

### 3.4 Ingest — `ingestDiscoveryRun(...)`

**Dispatch lives inside the ingest path, not the route** (audit): `run-callback` keeps passing
ids to `ingestScrapeRun`, which loads the row and claims it via the existing CAS
(`ingest_claimed_at`); after the claim, `mode === 'discovery'` branches to the discovery ingest.
Both paths share claim/idempotency/terminal-cost bookkeeping.

Per dataset item (short profile):
1. `canonicalizeLinkedinUrl` — drop items with no usable URL.
2. Drop if a contact exists for this user with that canonical `linkedin_url` OR
   `public_identifier` — **extract a shared helper** from `bulk-import.ts`'s inline
   `urlMap`/`pidMap` build (no shared helper exists today; don't duplicate the two-key logic).
3. Drop if in `suppressed_imports` (canonical-URL match only — that's the table's only key).
4. Upsert into `discovery_candidates` on `(user_id, linkedin_url)`: new → `status 'new'`;
   existing → bump `last_seen_at` + refresh headline/position/photo, **never touch status**.

Filter logic is a pure function `partitionDiscoveryItems(...)` with Vitest coverage.

### 3.5 Spend accounting fixes (audit A5 — three call sites + decoupling)

Current reserve math is `Math.max(1, contact_ids.length ?? 1) × (SCRAPE_UNIT_COST_USD[mode] ?? email)`
— a pending discovery run reserves **$0.01** (not $0 as Rev 1 claimed), ~10× under the real $0.10
page cost. Fix with `DISCOVERY_PAGE_COST_USD = 0.10` at **all three** sites:
1. `getMonthlySpendUsd` in-flight reserve (`scrape-service.ts:88-91`),
2. `sweepStuckRuns` timed-out cost estimate (`:281-283`),
3. the ingest catch fallback (`:448-449`).
Pattern: `r.mode === 'discovery' ? DISCOVERY_PAGE_COST_USD : Math.max(1, len) × unit`.

**Cadence decoupling (audit's top finding):** `sum_scrape_spend` has no mode filter, so discovery
spend would eat up to $2 of the cadence drip's $8 soft cap — discovery silently throttling
relationship maintenance. Decision: **decouple.** `triggerBatchScrape`'s affordability check
becomes `(CADENCE_SOFT_CAP_USD - (totalSpend - discoverySpend))`, with `discoverySpend` from
`sum_scrape_spend_mode`. The $10 **hard** cap stays global-total (one number for real money);
only the *soft* budget allocation is per-lane.

## 4. Scheduler — `POST /api/cron/discovery`

Copy the `cron/scrape-refresh` skeleton verbatim: QStash `Receiver` (current+next signing keys),
`maxDuration = 60`, 401 on bad signature, service client. Flow: select users
`.eq("status","active").eq("discovery_enabled", true)` (**not** `apify_enrichment_enabled` —
discovery is its own switch) → `triggerDiscoveryBatch` each. Trigger-only; ingest arrives via
webhook.

**Schedule:** QStash weekly — `0 10 * * 1` (Mondays 10:00 UTC), created via the region-pinned
QStash API (`qstash-us-east-1.upstash.io`), target
`https://www.careervine.app/api/cron/discovery` (www host, rule 29). Documented alongside the
other four schedules.

**Steady-state cost (Dawson-only, ~20 eligible companies):** 5 × $0.10/week ≈ **$2.17/mo**,
bounded by the $2 discovery soft cap, inside the $10 global cap.

### 5.2 Home — discovery digest card

### 5.3 Candidate APIs

- `GET /api/discovery/candidates` — session user's `new` candidates grouped by company
  (`?company_id=` filter for the company page).
- `POST /api/discovery/candidates/[id]/add` — build a schema-v1 people-record (mapper contract
  per audit A6):
  - `identity.name` = single string (join `firstName`+`lastName` or use `name`) — **no split**;
  - `identity.linkedin_url` = stored canonical URL; photo/headline/location in identity;
  - employment via `raw_profiles: [{ data: { experience: [{ companyName, position,
    companyLinkedinUrl: <companies.linkedin_url>, companyId: <linkedin_company_id if present> }] } }]`
    — threading the known company's LinkedIn identity makes `findOrCreateCompany` hit the
    existing row instead of inserting a name-ilike duplicate (the mapper reads employment
    **only** from `raw_profiles[].data.experience`; `identity.company/title` don't exist);
  - `pipeline.selected_contact: "SELECTED"` → clean `network_status: 'prospect'` (no warning);
  - `pipeline.found_by_searches: "discovery"` → `import_source: 'apify:discovery'` (a bare
    `'discovery'` is unreachable through the mapper);
  - → `importPeopleChunk` (default pipeline policy: suppression respected, create allowed;
    single record fine; photo download side effect accepted — best-effort 12s budget);
  - → mark candidate `added` + `added_contact_id` (from `PersonImportResult.contact_id`);
  - → fire `triggerEnrichOnSave(userId, contactId)` — it takes ids, checks
    `getApifyControls().enrichmentEnabled` + caps internally, never throws. (New contacts insert
    with `last_scraped_at = now`, but the no-email → email-mode path debounces on
    `scrape_failed_at`, so the enrich is not blocked.)
- `POST /api/discovery/candidates/[id]/dismiss` — status `'dismissed'`.

All session-authed via `withApiHandler`, service client for writes, ownership enforced.
